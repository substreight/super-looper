## 1. Intent

super-looper is a meta-skill whose **product is a refusal**. Its thesis — *"A loop is a goal plus a gate. The gate is the design"* — inverts the usual "let me build you an agent" reflex. Step 0's job is to talk you **out** of an autonomous loop, because most recurring tasks are better served by one good prompt, an interactive session, or a plain scheduler. Two genuinely original moves carry the whole project:

1. **Design the verifier before the loop body.** If you can't write a gate that automatically fails bad work *and* gates the deliverable (not the plumbing), you don't have a loop yet — and most tasks never get there.
2. **Autonomy is earned, not toggled.** An L0–L3 ladder where a validator computes the *earned ceiling* from gate rung + budget + scope fence + reversibility + a proven manual pass, and **refuses** anything above it.

Turning that philosophy into a mechanical refusal (`max_autonomy()`) is the single best idea in the codebase. The early version embodied it in well under 1,000 lines.

---

## 2. Where it's at vs the early version — earned or drift?

**One continuous history, not a real fork.** `super-looper` and the former `super-looper-audit` were always two checkouts of the *same* git repo; the repo has since been consolidated (the audit checkout became canonical `super-looper`; the prose original was snapshotted to `super-looper-archive-20260621`). `git log` is a single line v0.1.0 → v0.5.0. So the "two-tree divergence" risk is illusory. The *real* risk is the opposite: there is no separately-maintained lean core anymore — the minimal skill and the experimental lab share one tree, one version, one release.

**The evolution splits cleanly into earned and drift:**

- **Earned (the philosophy getting *more* disciplined):**
  - v0.2 — the autonomy dial (L0–L3) and `max_autonomy()`.
  - v0.3 — `DISCOVERY_REQUIRED` ("I don't know" becomes L0 evidence-gathering, not a guessed loop) and stricter L2/L3 requirements.
  These are direct, minimal extensions of the thesis.

- **Drift (a demonstration corpus that outgrew the thing it demonstrates):** everything from v0.4 on — the installable CLI, the case-study harness, the shadow verifier, the **2,051-LOC `repo_audit.py`**, the **688-LOC `remote_runner.py`** planner, the v0.6 promotion-packet taxonomy. The package is now **~5,504 LOC of Python**, of which the load-bearing thesis enforcement (`validate.py`) is ~655.

**Verdict (no fence-sitting):** the skill stays true to its thesis *in the prose* and *mostly in the validator*; the Python machinery has drifted into precisely the "elaborate machinery to look helpful" the skill warns users against. The largest module (`repo_audit.py`) is bigger than the entire v0.2 project and is a **subjective recommender with no gate on its own correctness** — exactly the shape the skill says disqualifies a loop ("Done is objective — not a matter of taste"). The v0.2 prose-skill-plus-validator was already the excellent, shippable product. v0.4–0.6 is a lab that should not be entangled with it.

Compounding the drift, the project currently fails several of its own "state on disk / done is objective" standards: a **dirty working tree** at HEAD, **`v0.4.0` was never tagged** (only `v0.3.0` and `v0.5.0` exist), and `pyproject.toml`/`__init__.py` still pin `0.5.0` while `CHANGELOG.md` and `README.md` document unreleased `0.6.0` behavior. A project preaching reproducibility can't currently name its own canonical state.

## 3. What's genuinely strong (preserve these)

- **The thesis is load-bearing and survived every version unchanged.** "Design the verifier before the loop body" and the default verdict of "no — use one good prompt" are intact from v0.2. The core was never diluted. ✓ traced
- **Earned-autonomy is *executable*, not asserted.** `max_autonomy()` (`validate.py:99-149`) mechanically refuses over-reach; the L3 branch genuinely requires an unattended trigger **and** `end_to_end` tool gate **and** reversible output **and** `proven_manual_pass` (lines 136-149). ✓ traced
- **The semantic lints are sharp and well-reasoned.** Self-grading is a hard error, unattended-without-gate is a hard error, unattended-without-budget is a hard error, parallelism-without-isolation is a hard error (`validate.py:388-459`). The reasoning text attached to each is genuinely educational. ✓ traced
- **The validator is a safe harness gate.** `validate()` never raises — it coerces every sub-object via `_d()` and short-circuits structural errors before the semantic pass (`validate.py:91-93, 498-506`). ✓ traced
- **The container sandbox is real.** When `isolation == "container"`, the emitted invocation is genuinely hardened: `--network none` (default), `--read-only`, `--cap-drop all`, `--security-opt no-new-privileges`, memory/cpu/pids limits, read-only mounts (`remote_runner.py:596-610`). ✓ traced
- **Intellectual honesty about limits.** The docs admit the validator "cannot confirm a gate that exists actually separates good output from bad," and `evidence.md` scopes its citations precisely and labels named techniques "mostly blog-evidenced." Rare calibration. • cited
- **The two schema copies are genuinely in sync**, loaded deterministically via `importlib.resources` with a source-tree fallback (`validate.py:33-52`). The feared drift does not exist. ✓ traced

### 4e. Design opinions (not defects — flagged as taste)

- `evidence_level` defaults to a strong tier on an absent/partial run dir rather than failing closed — wrong direction for a conservatism-first tool, though the orthogonal `ready_for_pr_claim` gate currently saves it. (`case_study.py`, summarize path) • cited
- `acceptable_verdicts` widening some eval scenarios to opposite conclusions (e.g. accepting both `AUTONOMOUS_LOOP` *and* `USE_SCHEDULER`) weakens discrimination — defensible per-case, but combined with the frozen results file it overstates accuracy.
- README/SKILL now front-load five CLI subsystems before the refusal-first message — the minimalism brand is undercut at first contact.

## 5. Complexity & elegance

**Elegant, keep as-is:** the no-raise validator harness (fuzz-tested with 400 mutations per the tests), the in-sync dual schema with deterministic load order, the default-deny scope fence, the once-computed evidence ladder, the genuinely-hardened container invocation.

**Over-built — cut or unify:**

- **The autonomy ladder is encoded in three places** — `max_autonomy()` (`validate.py`), the inline ladder in `design.build_spec` (`274-285`), and the schema enum prose. For a system that *gates on that ruleset*, three copies that can silently drift is the cardinal sin. Single-source through `max_autonomy`.
- **Two slightly different `MEASURABLE` regexes** (`validate.py:64` and `design.py:52`) will drift. Import one.
- **`repo_audit.py`'s imagined-candidate stanzas** (~480 lines of structurally identical prose literals) are a data table masquerading as control flow. Collapse to a dataclass table + one emit loop (~300 lines gone) with no lost fields.
- **The "shadow verifier template" abstraction is effectively fake** — a single hardcoded template keyed to one fixture's API. Elaborate machinery justified by one example.
- **The CLI eagerly imports the whole system** — `super-looper validate x.json` loads the SSH planner and the 2,051-LOC audit engine before argparse runs. Lazy per-subcommand imports keep the minimal path minimal.

## 6. The drift diagnosis — why it left center

Center is one sentence: **"A loop is a goal plus a gate; autonomy is earned via a validator that refuses over-reach."** The project drifted along three axes, and every confirmed defect maps to one of them:

1. **The validator stopped enforcing the thesis it preaches.** Gate quality is decoupled from the autonomy ceiling (#1), a typo'd cap passes (#2), vacuous gates compile (#3), and a guardrail attestation is fabricated by default (#4). The discipline moved into *prose* while the *enforcer* quietly relaxed.
2. **The project stopped passing its own gate.** Its CI cannot fail a skill regression (#15) and doesn't run the tests guarding its riskiest code (#16). The tool that says "maker ≠ checker" lets its maker grade its own homework.
3. **The minimal core got buried under a lab.** 5,500 LOC, five subsystems front-loaded ahead of the refusal-first message, a 2,051-LOC recommender with no gate on itself — the thing users adopt is no longer minimal, and the thing that made the project great (the lean skill + validator) is no longer separable.

The fixes in `IMPROVEMENT-TODO.md` are organized around reversing exactly these three vectors.
